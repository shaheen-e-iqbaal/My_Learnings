
In the HTTP request–response cycle, the body is ultimately transmitted as a stream of bytes. When the data being sent is **textual (like JSON, HTML, XML)**, it must first be converted from characters into bytes using a **character encoding** scheme. Character encoding defines how characters are mapped to bytes. The most widely used encoding on the web is UTF-8, which is a variable-length encoding capable of representing all characters across languages and symbols. For example, when a server sends a JSON response like `{"message": "Hello"}`, it first converts this string into bytes using UTF-8, and then sends those bytes over the network. The server informs the client about the encoding using the `Content-Type` header, such as:  
`Content-Type: application/json; charset=UTF-8`. The client uses this information to correctly decode the bytes back into readable text. It is important to note that character encoding applies only to textual data and not to binary data like images or PDFs (because we do encoding to get binary stream which we can transfer over internet. and this data is already in binary. so no need to do any encoding work for them).

Character encoding is necessary because computers can only transmit bytes, and simple encodings like ASCII are limited to 128 characters (mainly English letters, digits, and basic symbols). They cannot represent characters from other languages (like Hindi or Gujarati) or emojis. UTF-8 solves this problem by supporting all Unicode characters while remaining backward compatible with ASCII (i.e., standard English characters still take only 1 byte).

For **binary data** such as images (`image/png`), PDFs (`application/pdf`), or videos, no character encoding is needed because the data is already in byte form. The server directly sends raw bytes, and the client interprets them based on the `Content-Type`. For example, `Content-Type: image/png` tells the browser to treat the incoming bytes as a PNG image. In such cases, there is no `charset` involved.

After the data (text or binary) is converted into bytes, **content encoding** may be applied. Content encoding is used to compress the entire HTTP body to reduce payload size and improve network efficiency. Common compression algorithms include gzip, deflate, and Brotli (`br`). For example, the client can send a header like `Accept-Encoding: gzip, deflate, br` to indicate which compression formats it supports. The server may then compress the response using one of these methods and include a header such as `Content-Encoding: gzip`. This means the response body is compressed and must be decompressed by the client before further processing. The sequence of operations for a text response is: text → UTF-8 encoding → bytes → gzip compression → network transmission, and on the client side: receive → decompress → decode UTF-8 → reconstruct text.

Content encoding is optional and depends on server configuration and client support. It applies to the entire body (not per field or per part) and works for both text and binary data. However, formats like JPEG, PNG, and MP4 are already compressed, so applying gzip often provides little to no benefit and is usually skipped.

When both text and binary data need to be sent together in a single request or response, two common approaches are used. One approach is to encode the binary data into a text format like Base64 and include it inside JSON. For example: `{"image": "iVBORw0KGgoAAA..."}`. This works because JSON only supports text, but it increases the payload size by about 33%. The second and more efficient approach is to use `multipart/form-data`, where the HTTP body is divided into multiple parts using boundaries. Each part has its own headers and can contain either text (with charset encoding) or raw binary. For example, one part can be JSON metadata (`Content-Type: application/json; charset=UTF-8`) and another part can be a file (`Content-Type: image/png`) sent as raw bytes.

Finally, it is important to understand that in HTTP/1.1, only the body can be compressed using `Content-Encoding`; headers are sent as plain text. However, in HTTP/2 and HTTP/3, headers are also compressed using specialized mechanisms (HPACK and QPACK respectively). Overall, character encoding ensures that text is correctly converted between characters and bytes, while content encoding ensures that the byte stream is efficiently transmitted over the network.

-> In the early days of the internet, ASCII was used for character representation, but it supported only a limited set of characters (mainly English letters, digits, and basic symbols). As the internet grew globally, there was a need to represent characters from multiple languages and scripts. To solve this, Unicode was introduced, where each character is assigned a unique code point (for example, the rupee symbol has code point U+20B9). However, Unicode itself is just a mapping and cannot be directly transmitted over the network. To send this data, it must be encoded into bytes using encoding algorithms such as UTF-8 or UTF-16. Among these, UTF-8 is the most widely used because it is efficient and backward compatible with ASCII, and it is the default encoding in most modern systems like Spring Boot.

In HTTP communication, character encoding is applied only to the request or response body when it contains textual data. Headers do not use dynamic character encoding; instead, their format is defined by the protocol. In HTTP/1.1, headers are transmitted as ASCII-compatible text, while in HTTP/2, they are sent using a binary format and compressed using HPACK.

The complete flow on the client side is as follows: the client (browser, Postman, curl) receives a stream of bits from the OS socket. If the connection is HTTPS, these bits are first decrypted using TLS to obtain the original byte stream. Then, depending on the protocol version, headers are processed—either interpreted as ASCII text in HTTP/1.1 or decoded from binary frames and decompressed using HPACK in HTTP/2. Once headers are parsed, the client checks the `Content-Encoding` header to determine if the body is compressed (e.g., gzip) and decompresses it if necessary. After that, the client looks at the `Content-Type` header to find the charset (e.g., UTF-8) and uses it to decode the body bytes into a string. This decoding step converts the byte sequence into readable characters as per the specified encoding.

On the server side, the flow starts when the client sends an HTTP request, which reaches the server as a stream of bits through the network. These bits are received by the OS and passed to the TCP/IP stack, which provides a reliable byte stream to the server process (Tomcat). If the connection is HTTPS, Tomcat first performs **TLS decryption** to convert the encrypted bytes into the original HTTP byte stream.

Once Tomcat has the raw bytes, it begins parsing the HTTP request. Depending on the protocol version, headers are processed accordingly—either interpreted as ASCII-compatible text in HTTP/1.1 or decoded from binary frames and decompressed using HPACK in HTTP/2. Tomcat reads the incoming byte stream, detects the end of headers (`\r\n\r\n` in HTTP/1.1), and parses them into a structured request object.

After headers are parsed, Tomcat checks for the `Content-Encoding` header. If the request body is compressed (e.g., `gzip`), it decompresses the body to obtain the original byte sequence. Once decompression (if any) is done, Tomcat looks at the `Content-Type` header to determine how to interpret the body. If the body is textual (for example, `application/json; charset=UTF-8`), it uses the specified charset (UTF-8) to decode the byte stream into characters, effectively reconstructing the request content as a string or stream of characters. If the body is binary (like `application/octet-stream` or file upload), it is kept as raw bytes without any character decoding.

At this point, the request is handed over to the Spring Boot layer. Spring uses its **DispatcherServlet** to route the request to the appropriate controller. If the request contains a body (e.g., JSON), Spring uses **HttpMessageConverters** (like Jackson for JSON) to **deserialize** the request body. This means the JSON string (already decoded from bytes using UTF-8) is converted into a Java object (e.g., a DTO). For multipart requests, Spring processes each part separately, mapping files to `MultipartFile` and textual parts to appropriate objects.

Once the controller processes the request and returns a response object, Spring performs the reverse operation—**serialization**. The Java object is converted into a response format (e.g., JSON string) using the same message converters (like Jackson). This produces a textual representation of the response.

After serialization, Tomcat takes over again. If the response is textual, it is encoded into bytes using the specified charset (typically UTF-8, as indicated in `Content-Type`). If response compression is enabled and the client supports it (`Accept-Encoding: gzip`), Tomcat (or an upstream proxy) compresses the response body using the selected algorithm and sets the `Content-Encoding` header accordingly.

Finally, Tomcat constructs the full HTTP response by adding headers (formatted as ASCII-compatible text in HTTP/1.1 or binary frames in HTTP/2) and appending the response body bytes. If HTTPS is used, the entire response is encrypted using TLS. The resulting bytes are then sent back through the TCP/IP stack to the client.
